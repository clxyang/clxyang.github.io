Full-Stack React Application (Group)

Dorms at Brown

My team designed and coded a platform to navigate Brown University's 50+ on-campus housing options. We implemented TypeScript algorithms allowing users to search and filter for dorms by preferences such as room size or accommodations, and we integrated Firebase backend and Google user authentication so students can read/write reliable reviews on their dorms. See our GitHub repo here.

My role focused on designing the pages of the application in Figma as well as implementing the ranking and sorting algorithms in TypeScript.

Overview

Background & Objective

Every year during Brown's housing lottery for on-campus dorm selection, students are forced to rely on word of mouth and non up-to-date information to make crucial decisions surrounding their housing for the upcoming year. Our solution for making housing research more accessible was Dorms at Brown (D@B), a comprehensive dorm review website akin to Courses @ Brown (C@B).

This platform allows students to share detailed reviews and ratings on various aspects of dorm life, such as cleanliness, facilities, and location. By aggregating this data, our website also offers an overall "star" rating for each dorm, facilitating informed decision-making during the housing lottery process. With this resource, we sought to alleviate the stress of dorm selection by providing access to valuable dorm-related information for all students participating in the housing lottery.

Frontend Design Choices

Figma Designs

I aimed to bring a sense of familiarity to Dorms @ Brown by following the landing Page layout of Courses @ Brown. I wanted to make the flow very intuitive in terms of finding the dorms that apply to your class year, so I made these the main focus of the landing page. Additionally, I took inspiration from AirBNB's layout for a clean and comprehensive dorm listing design.

On the frontend, we used React.js with TypeScript and Tailwind CSS for styling. This combination allowed us to implement components that could be easily reused throughout the codebase.

Major Components

  1. Main: The main Dorms @ Brown page that holds all the other components.
  2. Dorms: Loads a set of dorms (either freshman housing, non-freshman housing, or off-campus housing), as well as a filter box and ranking box with different options for filtering/ranking the list.
  3. Dorm: Loads a particular dorm from a set of dorm by retrieving the corresponding document from a Firebase database and displaying the dorm info, rating, and reviews.
  4. OAuth: Authenticates a user to add a review only if they log in through their `@brown.edu` email address.
  5. Profile: Displays profile details once a user authenticates their account including name, email, and reviews written.
  6. Reviews: Handles reviews made by a user, including title, content, and ratings on cleanliness, facilities, and location.

Backend Design Choices

Technologies

The main technology used for our backend was Node.js (Express.js) to make our backend API endpoint calls. We chose this language because our frontend was using TypeScript, so we thought the integration would be more seamless if we used a JavaScript backend framework.

Backend endpoint calls:

  • Retrieve list of particular dorms based on dormtype (freshman housing, non-freshman housing, off-campus housing
  • Retrieve data of a particular dorm taking the dormtype and id
  • Update the rating statistics of a particular dorm
  • Delete a review

Firebase Firestore collections:

  • freshman-dorms
  • non-freshman-dorms
  • off-campus-housing
  • users

Implementing Filter/Sort Algorithms

I worked on implementing the filter and sort components to navigate a list of dorms. Filtering included room type (singles, doubles, triples, suites) and amenities (elevators, semi-private bathrooms, non-program housing). One difficulty I faced was the boolean logic behind different sorting options. I found that for room type, it made sense to include all dorms that included any of the selected room types (treating it as an "or"). However, for amenities, it made sense to only include dorms that met ALL selected amenities (treating it as an "and"). I tried to reflect this difference in the design as well by separating the filter options within the box.

Space-Time Complexity Choices

One consideration when designing our storage system was how to store reviews. We decided to add each review to two places -- both the particular dorm it was meant for and the user's collection. Therefore we could quickly access both the reviews array field of a particular dorm as well as reviews written by a certain user, instead of having to loop through all reviews to search for the ones written by a certain user.

Reflection

This was my first experience working on a full-stack application that used an API to integrate a backend database to a frontend display. It was definitely challenging to build up a complex dorm database and navigate trade-offs such as space-time complexity for our storage and algorithms.

With more time, future steps could include adding a search bar for dorms (originally in the design), as well as more precise filtering (eg. filtering for water stations).
